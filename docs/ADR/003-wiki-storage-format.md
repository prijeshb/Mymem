# ADR 003: Wiki Page Storage Format — Markdown over HTML

## Status: Accepted

## Context

MyMem stores LLM-generated wiki pages on disk. Two formats were considered:

- **Markdown (.md)** with YAML frontmatter — the current implementation
- **HTML (.html)** with meta tags or JSON-LD — the alternative

The decision matters because:
- LLMs read wiki pages on every `query` call (context injection)
- LLMs write wiki pages on every `ingest` call (LLM-generated output)
- Pages must carry structured metadata (title, domain, tags, sources, dates)
- Pages contain wikilinks (`[[Concept]]`) for the knowledge graph
- A React SPA renders pages in the browser

## Decision

**Store all wiki pages as Markdown (.md) with YAML frontmatter.** Do not convert to HTML for storage.

HTML is used only as a **rendering target** at the browser layer (React + marked.js).

## Rationale

### 1. Token Cost (Primary Driver)

Markdown reduces token usage by 60–90% compared to equivalent HTML. LLM calls are the dominant cost in this system. Every wiki page read by the LLM during `query` synthesis passes through the context window; smaller pages mean lower cost per query and more pages fitting in a single context.

### 2. LLM Generation Reliability

LLMs are trained on Markdown-heavy data (GitHub, Reddit, technical documentation). They produce syntactically correct Markdown more reliably than well-formed HTML. HTML generation risks unclosed tags, escaped entities in content, and malformed nesting — all of which degrade page quality silently.

### 3. Metadata Simplicity

YAML frontmatter is the standard for structured metadata in Markdown files:

```yaml
---
title: Transformer Architecture
domain: tech
tags: [ml, attention]
sources: [paper-x.md]
created: 2026-04-01
updated: 2026-05-29
---
```

Parsing is a single `yaml.safe_load()` call. HTML equivalents (JSON-LD, `<meta>` tags, microdata) require either a DOM parser or fragile regex and produce more tokens for the same data.

### 4. Wikilinks

`[[Related Concept]]` is unambiguous in Markdown and natural for LLMs to generate. HTML anchor equivalents require knowing the slug at write time and are harder to extract with simple regex.

### 5. Tooling Alignment

MyMem's existing stack is built around Markdown:
- `wiki_chunker.py` splits by Markdown headers natively
- `ingest.py` prompts the LLM to produce Markdown bodies
- `page.py` parses YAML frontmatter with `yaml.safe_load()`
- `api.py` returns raw Markdown to the frontend for client-side rendering

## Alternatives Considered

### HTML storage

**Rejected** because:
- 2–8× higher token cost per page in LLM prompts
- LLMs write malformed HTML more often than malformed Markdown
- Metadata extraction requires DOM parsing, not a single YAML call
- Wikilinks require custom HTML conventions
- Human-readable only in a browser, not in a terminal or text editor
- Diff output in git is unreadable for content changes

### Mixed (MD for content, HTML for export)

**Not needed now.** Export is a future feature. If export is ever added, convert Markdown → HTML at export time. Storage stays Markdown.

## Consequences

**Positive:**
- Lower inference cost per query (fewer tokens per page)
- Higher LLM write quality (fewer malformed pages)
- Simple metadata parsing (one function, no dependencies)
- Clean git diffs when pages are updated

**Negative:**
- Browser cannot render `.md` files natively — requires client-side parsing (marked.js, already in place)
- If a future feature needs HTML-specific semantics (ARIA attributes, semantic landmark elements), those must be generated by the frontend, not stored

**Risks:**
- LLMs occasionally generate HTML tags inside Markdown body — mitigated by `_strip_frontmatter()` and LLM prompt instructions to write clean Markdown
